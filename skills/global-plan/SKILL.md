# /plan — 설계 문서 작성

구현 전 설계 문서를 작성하고 `.claude/plans/`에 저장합니다.

## 목적

- 복잡한 기능 구현 전 아키텍처 설계
- 레이어 분리 및 의존성 방향 사전 설계
- 영향 범위 파악 및 위험 요소 사전 식별
- 사용자 확인 후 구현 진행 (명시적 작업 원칙)

## 프로젝트 컨텍스트 파악 (필수)

설계 전 프로젝트 루트의 `CLAUDE.md`를 읽어 다음을 확인합니다:
- 프로젝트 구조 및 레이어별 경로
- 기술 스택 및 통신 방식
- 기존 패턴 및 컨벤션

## 설계 문서 형식

다음 형식으로 `.claude/plans/{feature-name}.md` 파일을 생성합니다:

```markdown
# {기능명} 구현 계획

## 개요
- 목적:
- 영향 범위:
- 예상 복잡도: (낮음/중간/높음)

## 파일 변경 목록
| 파일 | 변경 유형 | 내용 |
|------|-----------|------|
| {경로} | 신규/수정 | {역할} |

## 레이어 설계
| 레이어 | 파일 경로 | 역할 |
|--------|-----------|------|
| Infrastructure | {경로} | 외부 시스템 연동 |
| Domain | {경로} | 비즈니스 로직 |
| Application | {경로} | 유즈케이스/훅 |
| Presentation | {경로} | UI 컴포넌트 |

## 통신/채널 설계 (해당 시)
| 채널/엔드포인트 | 방향 | 요청 타입 | 응답 타입 |
|----------------|------|-----------|-----------|

## 구현 단계
1. 타입 정의
2. Infrastructure 레이어 (서비스/API)
3. Domain 레이어 (비즈니스 로직, TDD)
4. Application 레이어 (훅/커맨드)
5. Presentation 레이어 (UI)

## 위험 요소 및 의존성
- 기존 코드 영향:
- 주의 사항:

## 검증 방법
- [ ] typecheck 성공
- [ ] lint 경고/오류 없음
- [ ] 테스트 통과
- [ ] 앱 동작 확인
```

## 사용법

```
/plan {기능명}
```

## 작업 절차

1. `CLAUDE.md` 읽어 프로젝트 구조 파악
2. 요청 기능 분석 (관련 파일 탐색)
3. 레이어별 구현 위치 결정
4. 통신/채널 설계 (프로젝트 통신 방식에 맞게)
5. 단계별 구현 계획 수립
6. `.claude/plans/{feature-name}.md` 저장
7. 계획 요약 보고 → **사용자 확인 대기**
8. 승인 후 `/implement`로 구현 진행
